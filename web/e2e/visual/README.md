# Visual Regression Baseline Guide

ADR-077 V-1 / V-2 / V-4. Playwright `toHaveScreenshot()` baselines
for visual regression detection. **Read before generating or
modifying baseline PNGs.**

---

## What is a baseline?

A PNG file in `__screenshots__/` that captures the expected visual
state for a `toHaveScreenshot()` assertion. Playwright auto-suffixes
the filename with `-{project}-{platform}` (e.g.,
`empty-viewport-chromium-win32.png` on Windows host).

Per ADR-077 V-D: matching tolerance is **maxDiffPixelRatio: 0.01**
(1% — anti-aliasing absorption).

---

## Cross-platform baselines (V-1 lock-in #3 / V-4-d)

PNG rendering differs across host OS:

| Host | Baseline suffix | Generated by |
|------|----------------|--------------|
| Windows | `-chromium-win32.png` | Local dev (V-1/V-2 commits) |
| Linux | `-chromium-linux.png` | CI ubuntu-latest (V-4 — pending) |
| macOS | `-chromium-darwin.png` | macOS dev (별도 sub-step) |

**현재 상태** (V-4 시점): Windows baselines 만 존재. Linux CI 첫 run
은 baseline missing 으로 fail 예상.

## First CI run procedure (V-4-b=(a) — fail + manual update)

When CI first runs Linux Playwright tests, visual specs will fail
with "A snapshot doesn't exist at ..." for each `*.visual.spec.ts`.
This is the **expected and documented** behavior per ADR-077 V-4-b.

**Procedure to add Linux baseline**:

### Option A: Run CI once + download artifacts

1. Open the failing CI run on GitHub Actions.
2. Download the `playwright-report` artifact (uploaded on failure).
3. Extract Linux PNGs from `test-results/**/test-failed-*.png`.
   Rename to match the snapshot naming convention.
4. Place in `web/e2e/visual/<spec-name>.spec.ts-snapshots/`.
5. Commit the new `-chromium-linux.png` files.

### Option B: Run Linux locally (Docker)

```sh
# From repo root
docker run --rm -it \
  -v "$PWD:/work" \
  -w /work/web \
  mcr.microsoft.com/playwright:v1.59.1-jammy \
  bash -c "npm ci && \
           cd ../crates/axia-wasm && wasm-pack build --target web --out-dir ../../web/src/wasm && \
           cd ../../web && npm run build && \
           npx playwright test --update-snapshots"
```

After completion, the `__screenshots__/` directory will have
`-chromium-linux.png` files. Commit them.

### Option C: Update via dedicated workflow (future V-3+)

A separate `workflow_dispatch` GitHub Actions workflow can run
`--update-snapshots` and create a PR with the new baselines.
**Not yet implemented** (V-3+ sub-step).

## Baseline update — when intentional changes happen

If a UX change is **intentional** (e.g., new outline color, new
viewport layout):

```sh
# Local — regenerate all visual baselines
cd web
npx playwright test e2e/visual/ --update-snapshots

# Verify the diff in the resulting PNG via git diff (or open the file)
# Commit the new baseline PNGs along with the code change
```

**Commit message convention**: include "visual baseline updated"
in the commit message body so PR reviewers know to inspect the PNG
diff specifically.

**PR review checklist** (per V-1 lock-in #4):
- [ ] PNG diff matches the intended UX change
- [ ] No accidental rendering drift (font / DPR / Three.js material)
- [ ] All affected platforms updated (`-win32`, `-linux`, etc.)

---

## Why git-track baselines? (V-C=(a) decision)

- **Reproducibility**: anyone can clone + run `npx playwright test`
  with identical reference.
- **PR review surface**: visual changes show up as PNG diffs in the
  PR — reviewers explicitly approve.
- **No external CDN dependency**: works offline, no auth.

**Trade-off**: git bloat. Currently 4 PNGs × ~644KB ≈ 2.6 MB.
At V-3 (multi-OS) this becomes ~8 MB. If exceeding ~50 MB, consider
Git LFS (separate ADR).

## Troubleshooting

### "Element is not stable" / "Animation in progress"

Visual tests use `animations: 'disabled'` (per ADR-077 V-D).
If a test still fails on CI but passes locally, the WASM rendering
may take longer to settle on Linux. Increase the
`page.waitForTimeout(500)` to 1000 in the spec.

### "Snapshot doesn't exist at ..." (CI Linux first run)

Expected. See "First CI run procedure" above.

### Diff > 1% on the same host

Either:
1. **Intentional UX change** — regenerate with `--update-snapshots`
2. **Unintentional drift** — investigate (Three.js update? Material
   change? Font fallback?) before updating
