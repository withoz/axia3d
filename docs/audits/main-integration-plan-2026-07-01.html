main 통합 계획 + 상세 시뮬레이션

2026-07-01 dry-run 완료 · 소스 무변경

목표: 실제 작동 엔진 adr-186/boundary-kernel-port(ADR-264)를 죽어 있는 main(ADR-163)의 정식 기준선으로 승격. 아래 시뮬레이션은 전부 read-only dry-run이며 어떤 브랜치도 변경/삭제/푸시하지 않았습니다.

결론 — 통합은 충돌 0, fast-forward. 예상보다 훨씬 안전합니다.

main은 adr-186의 직계 조상입니다(merge-base = main HEAD). 따라서 병합은 재작성·충돌 없이 490커밋을 그대로 얹는 fast-forward입니다. 위험은 “병합” 자체가 아니라 (1) 유실될 소수 발산 커밋의 취사선택과 (2) origin에 push하는 outward 행위 두 곳에만 있습니다.

Phase 0 안전망 (실행 전 필수, 되돌리기 보장)

어떤 것도 바꾸기 전에 현재 상태를 태그로 봉인. 이후 모든 단계가 이 태그로 100% 복구 가능.

git tag backup/adr110-line                 6afdbd2
git tag backup/adr142-line                 5a26fdf
# origin/main의 현재 위치도 기록 (push 롤백 기준)
git rev-parse origin/main > .integration-origin-main.sha

Phase 1 핵심 — main을 adr-186으로 fast-forward 충돌 0

이 한 단계가 “죽은 main” 문제의 90%를 해소합니다. 로컬에서만 수행 (push는 Phase 3).

# --ff-only: 충돌/머지커밋 없이 조상→후손 직진. 실패하면 안전하게 멈춤

Phase 3 origin 반영 outward · 승인 필요

로컬 검증 완료 후에만. origin/main을 490커밋 앞으로 이동 — public 저장소를 바꾸는 되돌리기 어려운 행위.

# fast-forward이므로 --force 불필요 (일반 push)
git push origin main
# 백업 태그도 원격에 (안전망 공유)
git push origin backup/main-pre-integration backup/adr186-snapshot

이후 로컬 adr-186/boundary-kernel-port는 main과 동일 커밋 → 브랜치는 지우고 backup/adr186-snapshot 태그로만 보존.

Phase 4 브랜치 대청소 (486 → ~5) + 재발 방지

# (a) 이미 adr-186(=main)에 포함된 로컬 98개 즉시 삭제
git branch | while read b; do b=${b#\* }; b=${b# };
  git merge-base --is-ancestor "$b" main 2>/dev/null && [ "$b" != main ] && git branch -D "$b"; done

# (b) superseded 스택 서브브랜치(adr-1xx/α·β·γ, claude/*) 일괄 삭제

git branch | grep -E '^(  (adr-1|claude/|audit/|ci/))' | xargs -r git branch -D

재발 방지 규율 (핵심)

  • 단일 기준선: 모든 실작업은 main에서 분기 → 완료 시 main으로 복귀. 장수 로컬 브랜치 금지.
  • 서브브랜치 금지: α/β/γ는 로컬 커밋으로만 쌓고 브랜치화하지 않는다. closure 시 squash/ff.
  • 브랜치 보호: GitHub에서 main 보호 규칙(직접 push 제한, CI green 필수) 설정.
  • 세션 브랜치 자동삭제: merge된 claude/*는 주기적 prune.

🚦 실행 승인 게이트

메모리 기록에 adr-186은 그간 “push 금지”로 표시돼 있었습니다. 본 통합은 그 정책을 사용자가 명시적으로 뒤집는 결정입니다. 따라서 아래 두 지점은 별도 확인이 필요합니다:

  • Phase 1(로컬 ff)까지는 안전 — 원하시면 바로 실행 가능(모두 로컬, 태그로 복구).
  • Phase 3(origin push)는 public 저장소를 490커밋 이동시키는 비가역 outward 행위 — 명시적 “push 진행” 승인 후에만.
  • Phase 2 provenance 살리기 여부 — “ADR-265로 살릴지 / 태그 보존 후 폐기할지” 결정 필요.

시뮬레이션 무결성. 본 문서의 모든 수치는 git merge-base --is-ancestor, git merge-tree --write-tree, git rev-list --count, git status --porcelain의 read-only 실행 결과입니다. 어떤 브랜치·태그·커밋·원격도 생성/변경/삭제하지 않았습니다. 위 명령 블록은 승인 시 실행할 계획일 뿐 아직 실행되지 않았습니다.
