3진짜 총체적 문제점 (심각도순)

P0  상태·통합 파편화 (§1)

  • 486 브랜치 · 10 워크트리 · main 490 커밋 뒤처짐. 단일 진실 상태 부재.
  • 실제 작동 엔진이 배포 기준선(main)에 한 번도 도달하지 못함 → 외부인·미래의 자신조차 “무엇이 최신인가”를 판단 불가.
  • 이것이 다른 모든 문제의 근원 — 진단 오류, 기능 연결 혼선, 검증 불신이 전부 여기서 파생.

P1  기능 연결 갭 (Pattern-12) + 실제 남은 기하 갭

  • Pattern-12(반복 패턴): Rust 엔진은 완비됐는데 WASM 래퍼·도구·카탈로그 배선만 빠져 “있는데 안 되는” 상태가 반복됨. (예: NURBS 편집 hook은 set_face_surface에 존재하나 WASM은 primitive만 노출.)
  • 실제 유효 갭 ①: 임의 닫힌 도형을 면 위에 그려 구멍/섬을 뚫는 면분할(imprint) — 여전히 미구현. “면 위 그리기 → cut” 워크플로우의 일반 케이스가 막혀 있음.
  • 실제 유효 갭 ②: extrude/cut 결과에 대한 범용 watertight production 게이트가 없어, 손상이 release 빌드에서 조용히 통과할 수 있음(ADR-264 β 미배포).

P1  ADR 프로세스 과부하

  • 264개 ADR + ADR당 α/β/γ 하위 브랜치 181개. 결정 하나에 문서·브랜치·회귀 세트가 대량 생성됨.
  • 추적성은 탁월하나 정리·병합·폐기 규율이 없어 순 부채가 단조 증가. 새 세션이 “무엇을 읽어야 하는지” 파악하는 비용이 급증.
  • CLAUDE.md LOCKED 섹션이 수천 줄 — 컨텍스트 예산을 상시 잠식.

P2  검증·관측 신뢰 부재

  • “회귀 통과 ≠ scene 경로 작동”이 반복 확인됨 — 단위/회귀 테스트가 실제 사용자 경로를 보장하지 못해 매번 수동 라이브 시연이 필요.
  • headless preview canvas가 0×0라 raycast 구동 드래그를 자동 검증 불가 → E2E 사각지대.
  • 외부 진단이 유령 상태를 겨눈 것도 결국 “관측 가능한 단일 상태”가 없어서.

4건강한 부분 (오해 방지)

  • 기하 커널: cut(boolean subtract + inward extrude)·slice·trim·drill·Boolean(곡면 포함)·NURBS 편집 5종 — 최신 엔진에서 실동작 확인.
  • 테스트 자산: 대량 Rust/vitest 회귀 통과(프로젝트 기록 기준 Rust ~2600 / vitest ~2470 pass), critical 0.
  • 커널 성숙도: ADR-027~264에 걸친 NURBS·SSI·surface citizenship이 상당히 깊게 구현됨. “stub이라 못 쓴다”는 재-패닉은 근거 없음.

5우선순위 권고

  1. 단일 진실 상태를 선언하고 main을 살린다. adr-186(최신)을 정식 통합 기준선으로 승격 — fast-forward 또는 squash로 main에 병합하고, 이후 “실작업은 반드시 main에서 분기 → main으로 복귀”를 규율화. 이것 하나가 P0의 90%를 해소.
  2. 브랜치 대청소. 병합 완료/폐기된 claude/* 64개 + 계단식 하위 181개를 일괄 정리(태그로 아카이브 후 삭제). 워크트리는 “활성 1 + main 1”만 남기고 나머지 제거. 목표: 486 → 30 이하.
  3. 범용 watertight 게이트 배포(ADR-264 β). extrude/cut/boolean 종료 시점에 verify_face_invariants + coincident-position non-manifold를 release에서도 강제하는 post-op assert. debug-only 검증을 production 게이트로 승격.
  4. 면분할 imprint 결단. “면 위 닫힌 도형 → 구멍/섬”은 CAD 핵심 워크플로우. 수요 없음으로 유예하지 말고 constrained Delaunay 기반 imprint를 정식 스프린트로 배정하거나, 명시적으로 “원형 hole 도구만 지원”을 제품 결정으로 못 박기.
  5. scene-path E2E 스모크 도입. 단위 회귀 통과에 의존하지 말고, cut/extrude/boolean/면분할의 실제 사용자 경로 + watertight 게이트를 한 번에 통과시키는 최소 E2E를 CI에 배치 → 수동 시연 의존 탈피.
  6. ADR 프로세스 다이어트. 하위 브랜치 α/β/γ는 로컬 커밋으로 남기고 브랜치화하지 않기. closure 시 문서만 병합. CLAUDE.md LOCKED는 요약 인덱스 + 상세는 개별 ADR 링크로 분리해 컨텍스트 예산 회복.
한 줄 요약. AXiA 3D의 엔진은 아프지 않습니다. 아픈 것은 “어느 것이 우리 엔진인가”를 답할 수 없는 상태입니다. main을 살려 단일 기준선을 세우고, 브랜치를 정리하고, watertight를 production 게이트로 올리는 것 — 이 셋이 “기능의 연결과 브랜치 과다” 문제의 핵심 처방입니다.
