AxiA 3D · BIM Interop

IFC Import/Export — 기반구축 & 확장 마스터 플랜

우리 analytic kernel(DCEL + AnalyticSurface/Curve + Shape/Xia/Material) ↔ IFC4.3 상호운용

2026-07-19 · 계획(audit + roadmap) · ADR-203 계속 + IFC import 신규 · 측정 기반

한 줄 요약. IFC export 기반(writer)은 이미 존재(axia-ifc β-1)하나 ① 실제 씬을 못 받고 ② 앱에 안 붙어 아직 동작하지 않음. IFC import는 placeholder("준비중"). → 권장 순서: export를 먼저 동작시키고(반쯤 지은 기반 활용) → analytic 확장 → import 착수.

1측정된 현재 상태

measure-first로 확인 — IFC는 greenfield가 아님(CLAUDE.md 포맷 표가 뒤처져 있었음).

영역상태근거 / 위치
Export 기반 (writer)완료 crates/axia-ifc (ADR-203 β-1). StepWriter + StepValue formatter SSOT + 결정적 22-char GUID + IfcFacetedBrep emit_box(min,max,name)/emit_unit_cube(). 진짜 IFC4X3 STEP-21. 21 tests.
Export — 실제 씬 geometry공백 axia-ifc가 axia-geo/axia-core 미의존 → primitive box만 emit, 실제 DCEL Mesh/Scene 못 받음.
Export — 앱 배선공백 axia-wasm 미노출 → 웹에서 호출 불가. 'IFC 내보내기' 명령/메뉴 없음.
Import (IFC)미착수 import-ifc 명령 + 메뉴 = placeholder ("IFC 가져오기는 준비중입니다"). 파서 0. MenuBar.ts:281, AxiaCommands.ts:315.
STEP-21 파서 (재사용 자산)있음 axia-foreign에 스키마 무관 Part-21 lexer/parser/resolver 완비(step_lexer.rs/step_parser.rs/step_resolver.rs). IFC도 STEP-21이라 이 front-end로 .ifc DATA를 그대로 파싱 가능 → IFC import는 entity classifier + promote 레이어만 추가.
런타임 배선 (공통)공백 axia-ifc·axia-foreign 둘 다 axia-wasm에 미포함 → 런타임 dead. 현 STEP/IGES import는 전부 OCCT.js(브라우저). IFC(export·import)는 WASM 바인딩 추가가 공통 선행조건.
기타 포맷 (참고)있음 Import: OBJ/STL/glTF/DAE/PLY/3DS/3DM/DWG = Three.js loader(디스플레이 전용, 편집 불가); DXF = Rust DCEL(편집 가능, Path B); STEP/IGES = OCCT.js + best-effort DCEL inject. Export: DXF/OBJ/GLTF/STL(전부 Three.js 씬 소스). Native .axia.
audit 결론. Import은 3개 병렬 파이프라인(Three.js 디스플레이 전용 / DXF→DCEL 편집 / OCCT STEP·IGES). Export은 4개 전부 Three.js 씬에서 추출 — DCEL→interchange 경로가 0 (β-1.5가 그 첫 경로). IFC는 export writer(β-1)와 import용 STEP-21 파서(axia-foreign)가 둘 다 in-repo에 존재하나 런타임에 안 붙어 있음.

2핵심 결정 (ADR-203 계승)

진짜 IFC emit

IFC4X3 엔티티명(IFCCARTESIANPOINT/IFCFACETEDBREP/IFCWALL). Revit/ArchiCAD/FreeCAD에서 열림 = BIM interop 목표 직결. STEP AP203명 혼용 금지.

결정적 byte-identical

순차 #N (insertion-order Vec), 결정적 GUID seed(geometry hash, OS-time 금지), formatter SSOT. 같은 모델 → 같은 .ifc 바이트.

StepValue format SSOT

REAL trailing-dot + sci-notation, STRING ''-escape + non-ASCII \X2\..\X0\, ENUM .IDENT., REF #N. 단일 모듈. import 파서도 이 SSOT의 역방향을 재사용.

Path Z atomic

각 sub-step = shippable 의미 단위(LOCKED #44), 별도 PR + 사용자 결재, 절대 #[ignore] 금지, 회귀 단조 증가.

3Export 로드맵 (동작 → 정밀)

ADR-203 β-1 완료 위에서 계속. 첫 목표 = 현재 모델을 .ifc로 저장 → 외부 BIM 도구에서 열림.

β-1.5실제 geometry FacetedBrep + 앱 배선~1주 · 첫 동작

axia-ifc가 axia-geo(Mesh/DCEL)·axia-core(Scene/Xia)를 의존하도록 확장. 씬의 모든 활성 face를 tessellate → IfcFacetedBrep(point + polyloop) emit(β-1 emitter 일반화). axia-wasm 바인딩 export_ifc() -> String(export_snapshot 답습) + WasmBridge wrapper + 웹 파일▸내보내기▸IFC 명령 + ExportUtils.downloadText(DxfWriter 다운로드 패턴). 이것이 엔진 DCEL을 interchange 포맷으로 내보내는 최초 경로(기존 4 exporter는 전부 Three.js 씬 소스) → 임의 모델 export 동작.

β-2IfcAdvancedBrep + analytic surfaces~1주

Plane/Cylinder/Sphere/Cone/Torus/BSpline/NURBS surface → IfcPlane/IfcCylindricalSurface/... + IfcAxis2Placement3D/IfcDirection + IfcEdgeLoop/IfcOrientedEdge. tessellation-loss 없는 진짜 곡면 export (우리 kernel의 강점).

β-3AnalyticCurve → IFC edge curves~1.5주

Line/Circle/Arc/BSpline/NURBS → IfcLine/IfcCircle/IfcTrimmedCurve/IfcBSplineCurveWithKnots/IfcRationalBSplineCurveWithKnots. Bezier→BSpline knot 합성. ADR-036 P21 매핑표의 역방향.

γXia → IfcWall + Material + Representation~0.5주

Xia(부재) 열거 → IfcWall + IfcMaterial(name) + IfcProductDefinitionShape/IfcShapeRepresentation. (element-type 세분은 origin_hint 후속 — MVP는 전부 Wall.)

δSpatial hierarchy~0.5주

Project→Site→Building→Storey→Element + IfcRelContainedInSpatialStructure/IfcRelAggregates. 단일 storey MVP.

ε외부 IFC 검증 + 시연~1주

IfcOpenShell/Revit/FreeCAD round-trip, corpus 검증, 사용자 시연(ADR-087 K-ζ). β 이전은 자체 well-formedness(모든 #N 해소·닫힌 graph·좌표 정확) 검증.

4Import 로드맵 (신규 — 별도 ADR)

좋은 소식 — 파서 절반이 이미 있음. IFC는 물리적으로 STEP-21. axia-foreign의 Part-21 lexer/parser/resolver는 스키마 무관이라 .ifc DATA를 그대로 파싱함(신규 파서 불필요). IFC 전용으로 필요한 건 entity classifier(IFC4X3명 인식) + geometry promote(brep→DCEL / analytic surface) 레이어뿐 — classify_*_entity 패턴 답습. (axia-foreign은 곡선/곡면만 promote하고 brep topology는 아니라 brep→DCEL은 신규.) → import 난이도·추정이 애초 예상보다 낮음.
I-1STEP-21 front-end 재사용 + WASM 배선~0.5주 · 신규 거의 0

axia-foreign의 step_lexer/step_parser/step_resolver를 그대로 사용(IFC DATA 파싱 가능). 공통 선행조건 axia-wasm 바인딩 추가(현재 두 Rust 크레이트 모두 런타임 dead — export β-1.5와 공유하는 배선 작업).

I-2IFC 엔티티 인식 + graph 해소~1.5주

IFC4X3 엔티티 인식(IFCFACETEDBREP/IFCPOLYLOOP/IFCCARTESIANPOINT/IFCADVANCEDBREP/surface·curve). #N 참조 그래프 해소 + 순환/누락 방어.

I-3geometry → DCEL~2주

IfcFacetedBrep → verts/faces → Mesh(spatial-hash dedup, LOCKED #5). IfcAdvancedBrep/surface → AnalyticSurface promote(ADR-036 P21 매핑 재사용). winding/manifold 정합(ADR-007).

I-4spatial/시맨틱 → Scene~1주

IfcWall → Xia(부재), IfcMaterial → MaterialLibrary, spatial → group. owner-ID 매핑(ADR-037 P22).

I-5배선 + placeholder 대체 + 시연~0.5주

axia-wasm 바인딩 + 웹 파일▸가져오기▸IFC(현 "준비중" 대체) + FileImporter 통합 + 사용자 시연.

5스코프 / Out-of-scope (MVP)

In-scope

  • Export: FacetedBrep + AdvancedBrep(analytic surface) + edge curve + Xia→IfcWall + Material(name) + 단일 spatial + 단위/OwnerHistory
  • Import: FacetedBrep + (가능 시) AdvancedBrep → DCEL/analytic + IfcWall→Xia + Material
  • 결정적 byte-identical export, 진짜 IFC4X3, self-validation + 외부 도구(ε)

Out-of-scope (후속 ADR)

  • Assembly/composition hierarchy, Drawing views
  • PMI/GD&T/annotation
  • Material cost·thickness (필드 부재 → IfcCostValue 미지원)
  • element-type 세분(origin_hint 부재 → MVP 전부 IfcWall)
  • IfcRectangularTrimmedSurface 등 trim loop 시맨틱

6Lock-ins · 리스크

Lock-ins

  • 신 crate 분리(axia-ifc export / import 파서는 별 모듈), axia-foreign import 보존
  • 결정적 byte-identical + StepValue SSOT + 진짜 IFC4X3명
  • ADDITIVE 신규 scope — LOCKED 역전 0 (ADR-035 §P20.A IFC defer 충족)
  • Path Z atomic + 사용자 결재 + 절대 #[ignore] 금지

리스크

  • R1 formatter 정확성(REAL/STRING) → SSOT + 단위 테스트
  • R2 결정성 → DCEL 순회 결정적(collect_loop_hes)
  • R3 IFC 시맨틱 검증 gap → ε 외부 도구
  • R4 import entity-graph 복잡도(순환/버전 IFC2x3 vs IFC4) → 방어적 파서
  • R5 geometry 정밀도(round-trip <1e-3mm)

7권장 착수 순서 & 결재 포인트

  1. β-1.5 (Export 실동작) — 반쯤 지은 기반을 shippable 기능으로. 첫 사용자 가치(모델→.ifc→Revit). 첫 결재
  2. β-2 → β-3 → γ → δ → ε — export 정밀·완성 (ADR-203)
  3. Import I-1~I-5 — export writer/SSOT 성숙 후 착수(파서가 formatter 역방향 재사용). 별도 ADR.
다음 결재: 이 계획 승인 + β-1.5 착수 여부. (β-1.5 α spec를 별도 ADR로 쓸지, ADR-203 amendment로 흡수할지도 결정.)
