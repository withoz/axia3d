AXiA 3D — 종합 구현 계획

2026-07-01 기준: clean repo E:\AXiA3D @ ADR-264

이번 총체 점검에서 코드로 검증된 실제 갭만을 대상으로 한 go-forward 로드맵. 외부 보고서의 stale 주장(cut 부재 등)은 제외했고, 실증된 문제(면 imprint 미구현 · watertight 게이트 debug-only · 기능 연결 갭 · 프로세스 규율)에 집중합니다. 모든 단계는 사용자님 방법론을 따릅니다 — Path Z atomic · 시연 게이트 · 회귀 자산(#[ignore] 금지) · LOCKED/메타-원칙 준수.

우선순위 한눈에. Phase 0(기반)은 나머지 전부의 전제 → 먼저. Phase 1(CAD-core 갭)이 제품 가치의 핵심. Phase 2·3은 병행 가능. Phase 4는 장기. 아래 순서는 권장안이며 재배치 가능합니다.

§0공통 방법론 (모든 단계 적용)

  • Path Z atomic — 각 작업을 α(spec) → β(engine) → γ(bridge) → δ(UI) → ε(E2E) → ζ(closure) sub-step으로 분해, 각 sub-step은 독립 commit + 회귀 통과 후 진행.
  • 시연 게이트 — 회귀 통과 ≠ scene 경로 작동. 각 Phase 종료 시 브라우저 실동작 시연(fresh reload + eval)으로 최종 확인. (메모리: 시연이 4개 회귀를 잡은 전례.)
  • 회귀 자산 — 신규 기능마다 Rust/vitest/Playwright 회귀 추가, 절대 #[ignore] 금지.
  • LOCKED/메타-원칙 준수 — 특히 #14(면은 닫힌 경계로부터 유도), #15(headless ≡ tool path 의미 등가), ADR-007(winding), ADR-046 P31 #4(메뉴 additive only).
  • 새 ADR 부여 — 각 트랙은 신규 ADR-265+ 번호 (구 repo와 충돌 없음, 깨끗한 연속 번호).

Phase 0기반 안정화 P0 · 선행 필수

새 repo가 “한 방에 빌드·CI·기여 가능”해야 이후 모든 작업의 마찰이 사라짐. 앞선 검증에서 드러난 마찰을 근본 제거.

0.1 — fresh install 근본 수정 (ADR-265)
규모: S (1일) · 파일: packages/axia-action-catalog/package.json, (선택) 루트 package.json
현재 cd web && npm install이 실패 — @axia/action-catalog의 prepare:tsc가 monorepo 루트 부재로 tsc를 못 찾음.
  • 옵션 A (권장): 루트 package.json에 npm workspaces: ["web","packages/*"] 선언 → 단일 npm install로 전 패키지 devDeps 설치 + prepare 정상.
  • 옵션 B (경량): catalog prepare를 prepare: "tsc || exit 0" + web postinstall에서 catalog 빌드 보장.
수용: 깨끗한 clone에서 npm install && npm run build:all 무오류 + Windows EPERM 롤백 없음.
0.2 — CI 되살리기 (ADR-266)
규모: S~M (1~2일) · 파일: .github/workflows/{build,ci,mcp}.yml
6개 workflow가 있으나(구 repo에서 red 이력, ADR-102 계열) 새 baseline에서 green 보장 필요.
  • rust-test / vitest / playwright job이 clean repo에서 통과하도록 경로·캐시·의존성 갱신 (0.1 workspace 반영).
  • MCP capability handler가 최신 _as_shape API와 정합하는지 확인(구 ADR-102 Track A 잔재 점검).
  • 기준선 health 재측정: Rust ~2600 / vitest ~2470 (메모리 기록과 대조).
수용: main push 시 3 job 전부 green. 배지 README 반영.
0.3 — 단일 기준선 규율 확립 (프로세스)
규모: S (0.5일) · GitHub 설정 + 문서
브랜치 재폭발 방지 (구 repo 486개의 교훈):
  • GitHub branch protection: main 직접 push 제한 + CI green 필수 + linear history.
  • 서브브랜치 금지 규약: α/β/γ는 로컬 커밋으로만, 브랜치화 안 함. closure 시 squash/ff.
  • 세션 claude/* 워크트리 주기적 prune. 활성 워크트리 = “작업 1 + main 1”만.
  • CONTRIBUTING.md에 규약 명문화.
수용: main 보호 규칙 활성 + CONTRIBUTING.md 커밋.

Phase 1CAD-core 실제 갭 P1 · 제품 핵심

총체 점검이 코드로 확인한 두 실증 갭 + extrude/cut 완성. 이것이 “블렌더보다 쉽고 스케치업보다 정확한” 약속의 본체.

1.1 — Universal Watertight Production Gate (ADR-267)
규모: M (3~5일) · 파일: mesh_invariants.rs:364, operations/{create_solid,boolean,face_split}.rs, axia-wasm, Toast
현재 debug_verify_invariants는 #[cfg(debug_assertions)] — release에서 손상이 조용히 통과. ADR-264 β의 미배포 게이트를 완성.

Path Z

  • α spec: 게이트 정책(어떤 op 종료 시 무엇을 검사) + 실패 시 계약(typed error + 자동 rollback + 사용자 다이얼로그).
  • β engine: verify_face_invariants + coincident-position non-manifold 카운트(ADR-264 D3 lock-in) + solid-scope post-op assert를 release 경로로 승격. degenerate/NaN/비-watertight 감지.
  • γ bridge: verifyVolumeIntegrity(faceIds) → Report WASM export + 각 volume op 종료 시 자동 호출.
  • δ UI: 실패 시 Toast + (ADR-097 패턴) 복구 다이얼로그 [Undo]/[강등]/[수동수정].
  • ε E2E: extrude/cut/boolean/split 후 getStats + integrity 검사 real Chromium.
수용: 손상 유발 시나리오가 release에서 silent pass 0건 + 정상 op 회귀 무영향. 시연: 실제 cut 후 watertight 확인.
1.2 — Face Imprint: 면 위 닫힌 도형 → 구멍/섬 (ADR-268) flagship
규모: L (2~3주) · 파일: face_split.rs:264/1848(“future” 마커), mesh.rs, tools, WASM
실증된 최대 갭. 면 boundary 위가 아닌 내부에 닫힌 도형을 그렸을 때 inner face + outer ring으로 분할하는 경로가 미구현 (face_split.rs:264 "(future)"). “면 위 그리기 → cut” 워크플로우의 일반 케이스를 차단.

Path Z

  • α spec: 메타-원칙 #14 정합 설계. closed inner loop → 1 ring(outer+hole) + 1 disk(inner) 또는 별개 sub-face (ADR-021 P7 / ADR-101 mergeCoplanarContaining 재사용).
  • β-1 engine (analytic 우선): inner loop가 원/사각 등 kernel-native면 기존 punch/hole 합성 경로 확장 (ADR-089 closed-curve citizenship 활용).
  • β-2 engine (일반): 임의 폴리곤 inner → constrained Delaunay 또는 combined-perimeter ring 합성. ADR-006 Phase F/G 자산 재사용.
  • γ/δ bridge + DrawTool: 면 hit 시 그 면을 host로 삼아 imprint(메모리: “host from world point, no stale id” 원칙).
  • ε E2E + #15 의미 등가: headless _as_shape 경로 ↔ tool 경로가 동일 hole 결과.
수용: 면에 원/사각/자유 폐곡선 → 구멍 생성 + invariants 0 + push/pull로 관통 가능. LOCKED #1/#12 회귀 유지.
1.3 — Extrude/Cut 완성 + 단일화 (ADR-269 ~)
규모: M~L · 파일: create_solid.rs, push_pull.rs, tools · 메모리 “완벽한 extrude” 트랙 연속
기존 트랙(#1 테이퍼 ~ #5 곡면 Phase0)은 완료. 잔여를 clean repo에서 마무리:
  • P1 곡면 cut — 곡면 벽에 sketch한 영역 → pocket/관통 (현재 곡면 직접 drill 미지원, boolean 우회만).
  • P2 곡면 boss — 곡면 위 돌출 (foundation 충족 후).
  • #6 separated-disk — 분리된 disk extrude 갭.
  • Extrude ↔ Push/Pull 의미 단일화 — 두 진입점의 SSOT 정리(ADR-080 dimension-dispatch 정합).
수용: 각 항목 real Chromium 시연 + watertight 게이트(1.1) 통과.

Phase 2기능 연결 + 검증 인프라 P2 · 병행 가능

2.1 — Feature-Connection 감사 (Pattern-12 해소, ADR-270)
규모: M · 산출물: 갭 인벤토리 + 우선 배선
“Rust 엔진 완비, WASM/도구/카탈로그 배선만 빠짐”이 반복. 체계적 감사로 목록화 후 배선.
  • 모든 pub fn engine op ↔ WASM export ↔ ActionCatalog ↔ 도구/메뉴 매핑 표 생성 → 미배선 op 목록.
  • 예: set_face_surface는 NURBS 수용하나 WASM은 primitive만(메모리) → 곡면 live 배선.
  • 우선순위: 사용자 가치 높은 것부터 (곡면 편집 live, NURBS 패널 슬라이더 등).
수용: 감사표 커밋 + 상위 N개 배선 + ActionCatalog SSOT 정합 회귀.
2.2 — Scene-Path E2E Smoke (ADR-271)
규모: M · 파일: web/e2e/ (ADR-075 인프라 재사용)
단위 회귀가 scene 경로를 보장 못하는 문제(“회귀통과 ≠ 작동”)의 구조적 해소.
  • 핵심 워크플로우 스모크: draw → extrude → cut(1.1) → imprint(1.2) → boolean, 각 단계 watertight 게이트 통과까지 real Chromium.
  • CI web-e2e job에 편입 → 수동 시연 의존 감소.
  • headless canvas 0×0 한계(메모리)는 engine lifecycle 직접 호출 + unit로 보완.
수용: 스모크가 CI green + 의도적 손상 주입 시 red(게이트 작동 증명).
